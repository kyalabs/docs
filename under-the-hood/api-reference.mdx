When it ships, the API will read a delivered study and its data package programmatically: to ingest into a warehouse, drive a dashboard, or chain studies into downstream pipelines. This page describes the planned shape.

## Design principles

- **Read-only at v1.** Study submission and configuration go through the platform; the API is for reading delivered work.
- **One auth model.** Workspace-scoped keys, rotatable, issued from the workspace admin surface. SSO-tied keys arrive with the [access & admin](/offered/engagement-lifecycle#access-admin-and-continuity) layer.
- **Same data, same schema.** What the API returns is byte-equivalent to what the [cloud-storage handoff](/under-the-hood/data-outputs) delivers.
- **Integrity-verifiable.** Every response is designed to carry the same checksum the data package carries, verifiable on receipt.

## Planned read surface

The v1 design is a small set of read endpoints over a delivered study: its record, its trips, its recommendations and non-recommendations, its verdict, and its sources, plus a checksum manifest. Responses stream as newline-delimited JSON, identical to the objects in the [data package](/under-the-hood/data-outputs).

Exact paths, the auth scheme, per-key scopes, and rate limits are part of the v1 contract finalized before launch, so an integration planned against the shape above doesn't harden against details that may still move.

## What's planned

| Capability | Status |
| --- | --- |
| Read delivered studies, trips, recommendations and non-recommendations, the verdict, and sources | **v1, committed.** |
| Webhook on study delivery (a notification carrying the study id and a manifest reference) | **Planned.** |
| Submit a study from a saved mission | **Planned (v2).** |
| Compose a new mission via API | **Planned (v2).** |
| Real-time stream of in-flight trips | **Not planned.** A study stays immutable until it is delivered. |

See [the roadmap](/roadmap) for where each one sits.
