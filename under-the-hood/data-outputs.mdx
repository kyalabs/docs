A delivered ShoppiPanel study returns two artifacts: a **report** (the read) and a **structured extract** (the machine-readable layer behind it). This page covers the structured extract.

The extract is the machine-readable form of everything in the report. The report renders the extract; if a number is in the report, it's in the extract.


<Note>
  **Maturity stamp.** The **shape, field names, and semantics on this page are live today.** They match the extract the engine emits per run (snake_case, schema `standard_extract.v0`). What is **rolling out** is the *self-serve export interface*: today the extract is handed to you operator-mediated (white-glove). A self-serve cloud-storage handoff is **planned for integration** (see [Transport](#transport)). Confirm the integration path with your kya labs contact before you wire it into a pipeline.
</Note>

## Delivery shape

The extract is **one pretty-printed JSON file** per run: not gzipped, not newline-delimited, not split across multiple objects. One run produces one `standard_extract.v0.json` plus a companion `fidelity_receipt.v0.json` (the same fidelity receipt is also embedded under the extract's `fidelity_receipt` key).

| Property | Value |
| --- | --- |
| **Format** | A single pretty-printed JSON document (UTF-8). Keys are snake_case. |
| **Schema handle** | `standard_extract.v0`, carried on every extract as the top-level `schema_version` |
| **Files per run** | `standard_extract.v0.json` (the extract) · `fidelity_receipt.v0.json` (the receipt, also embedded) |
| **Content hash** | A single **run-level content hash** over the package: `sha256:<hex>`, content-addressed. **This is a per-run hash, not a per-object signed manifest.** |

A run's extract is immutable once produced. Re-runs produce a new `run_id`; never a mutation of the old one.

### Transport

| Path | Status |
| --- | --- |
| **Operator-mediated (white-glove) handoff** | **Live today.** Your kya labs contact delivers the extract for a delivered run. |
| **Self-serve cloud-storage handoff** (into a bucket you designate) and **in-product structured-data export** | **Planned for integration; rolling out.** Until it lands, export runs with us in the loop. See [the roadmap](/roadmap). |

## Top-level shape

The extract is a single object. These are its top-level keys, as the engine emits them:

  </Card>

  <Card title="input" icon="book-open">
    The redacted operator brief that framed the run: `focus_brand`, `category`, `research_question`, `study_type`, `competitive_set_posture`.

  <Card title="population" icon="layer-group">
    The count ledger: how many cells were planned, completed, did not run, and are metric-eligible. Field-level breakdown [below](#population).
  </Card>

  <Card title="trips" icon="route">
    One record per trip: the receipt for one agent's run. The atomic unit of evidence (steps, evidence refs, status, eligibility).
  </Card>

  <Card title="entities" icon="users">
    The brands / products the study reasons over, derived from the manifest, the trips, and the filtered recommendations.
  </Card>

  <Card title="recommendations" icon="circle-check">
    The recommendations the panel surfaced, each backed by source cards. Only recommendations whose evidence refs all resolve to a source card are carried.

  </Card>

  <Card title="non_recommendations" icon="split">
    The companion set to `recommendations` (brands the panel reasoned over but did not recommend).
  </Card>

  <Card title="source_cards" icon="globe">
    One record per fetched source: the access posture, the request, the response signature, the provenance.
  </Card>

  <Card title="citations" icon="file-chart-column">

  <Card title="caveats" icon="triangle-exclamation">
    The qualifiers attached to the read: what to discount and why.
  </Card>

  <Card title="claim_boundary" icon="shield-check">
    The claim boundaries from the compiler: what the study does and does not assert.
  </Card>

  <Card title="verdict" icon="gauge">
    The publish verdict and any blockers, reconciled against the population.
  </Card>

  <Card title="telemetry" icon="wrench">
    Run telemetry: trip and source-card rollups, cost ledger.
  </Card>

  <Card title="fidelity_receipt" icon="microscope">
    The fidelity metrics and any research-grade flags. Also emitted as a standalone `fidelity_receipt.v0.json`.
  </Card>

  The extract also carries an envelope: `schema_version` (`standard_extract.v0`) and `generated_at` at the top level alongside the keys above.

## Schema reference

<Note>
  The shape below is **live today.** It matches what the engine emits. Field additions are non-breaking; field removals or type changes ship under a new schema handle. The current handle is recorded on every extract as the top-level `schema_version`.
</Note>

### Envelope + `identity`

```json
{
  "schema_version": "standard_extract.v0",
  "generated_at": "2026-06-30T14:22:00Z",
  "identity": {
    "run_id": "run-023",
    "study_id": "stu_2026q2_crm",
    "report_slug": "crm-buyer-panel",
    "methodology_version": "panel.2026-Q2.r3",
    "methodology_version_sha": "a1b2c3...",
    "compiler_version": "standard_extract.v0",
    "generated_at": "2026-06-30T14:22:00Z"
  }
}
```

### `input`

```json
{
  "input": {
    "focus_brand": "acme",
    "category": "CRM for small teams",
    "research_question": "Which CRM should a 20-person team pick?",
    "study_type": "competitive-panel",
    "competitive_set_posture": "named-set",
    "redaction_status": "redacted_operator_brief"
  }
}
```

### `population`

The count ledger. **`blocked` is an internal COUNT field for cells that never ran or failed the pipeline. Read it as "did not run."** It is not a [reach posture](/insights/reading-a-report/reach); that lives on each source card as *reached / access-limited*.

The schema splits completed cells into two fields, which sum to the report's `completed` total: `completed` = `completed_detailed` + `completed_thin`.

```json
{
  "population": {
    "planned": 100,
    "completed_detailed": 62,
    "completed_thin": 16,
    "blocked": 18,
    "insufficient": 4,
    "metric_eligible": 78,
    "source_backed": 78,
    "cells_with_evidence": 80
  }
}
```

| Field | Meaning |
| --- | --- |
| `planned` | Cells the run intended to dispatch. |
| `completed_detailed` | Source-backed cells minus thin ones: full receipts. |
| `completed_thin` | Source-backed but capture-degraded cells. |
| `blocked` | **Did not run / incomplete.** Cells that never ran or failed the pipeline. Internal count name; not a reach posture. |
| `insufficient` | Planned cells that neither completed source-backed nor failed outright. |
| `metric_eligible` | Cells eligible to compute a metric. |
| `source_backed` | Cells that fetched a page *and* produced a usable recommendation. |
| `cells_with_evidence` | Cells that fired and returned at least one usable evidence ref. |

### `trips[]`

```json
{
  "trip_id": "trp_01H...",
  "status": "pass",
  "agent": { "vendor": "<agent-vendor>", "model": "<model-class>", "retrieval_mode": "browse-native" },
  "persona_cohort": "smb-it-buyer",
  "mission": "Which CRM should a 20-person team pick?",
  "steps": [ /* ordered step records — searches, fetches, citations */ ],
  "evidence_refs": ["src_01H...", "src_01H..."],
  "capture_state": "capture_ok",
  "eligibility": { "compiler_eligible": true },
  "artifact": "The agent's verbatim final answer text.",
  "completed_at": "2026-06-30T13:45:12Z"
}
```

### `source_cards[]`

Per-card [reach posture](/insights/reading-a-report/reach) is **reached / access-limited**.

```json
{
  "source_id": "src_01H...",
  "cell_id": "trp_01H...",
  "url": "https://example.com/products/checkout",
  "access_posture": "access-limited",
  "access_limited_reason": "bot-challenge",
  "status": 403,
  "provenance": "fetch",
  "title": "Checkout — Example",
  "gist": "...",
  "fetched_at": "2026-06-30T13:44:50Z"
}
```

### `citations[]`

```json
{
  "citation_id": "src_01H...:citation",
  "source_id": "src_01H...",
  "url": "https://example.com/products/checkout",
  "title": "Checkout — Example",
  "provenance": "fetch",
  "snippet": "...",
  "trip_ids": ["trp_01H..."]
}
```

### `fidelity_receipt`

The fidelity metrics and any research-grade flags. Quality signals here [flag, they do not hold](/evidence/honesty-layer): a run with a breached floor still ships, stamped with the flag.

```json
{
  "fidelity_receipt": {
    "metrics": {
      "planned": 100,
      "source_backed_count": 78,
      "source_backed_coverage": 0.78,
      "degraded_to_memory_count": 3
    },
    "blockers": [],
    "warnings": [],
    "artifact_ref": "compiled/fidelity_receipt.v0.json"
  }
}
```

Full field-level definitions ship alongside the extract; confirm the integration path with your kya labs contact before you wire it into a pipeline.

## Stability guarantees

| Change type | Versioning |
| --- | --- |
| New optional field | Non-breaking. Same `schema_version`. |
| New enum value (e.g., a new `access_limited_reason`) | Non-breaking. Documented in the changelog. |
| Removed field, type change, or semantic change | Breaking. New `schema_version` handle, with a migration window. |

The current schema handle is `standard_extract.v0`, recorded on every extract.
