<Note>
  **For technical evaluators.** How a measurement stays comparable as the models change. This is the answer to *"is this a real metric, or a snapshot?"*
</Note>

AI models update constantly. If the thing doing the shopping changes, how can a measurement taken in June be compared to one taken in September?

The panel is a **versioned instrument**, not a dial. Its composition is locked per study, every run is stamped with exactly what produced it, and any change to the instrument is a versioned event rather than a silent one.

## What a panel is

A **panel version** is a locked tuple of three things:

- **Composition:** which model families, how many trips each, and across which lanes (browse and search-native; [Two Lanes](/how-it-works/two-lanes) covers what those modes are and how each is captured). The families are **GPT, Gemini, Claude, and Perplexity Sonar** (the search-native lane). The panel-of-record is a 25-trip mixed panel, composed to track the live consumer-agent mix: the largest share of trips on the most-used family (a plurality, not a majority), the rest split across the others, with Perplexity Sonar reading the way people ask rather than browse. Smaller studies run a lighter default composition; the study always declares which one it ran on. A 25-trip panel is a **directional instrument**, not a statistical-significance test (see [Headline and verdict](/insights/reading-a-report/headline-and-verdict)); where a question needs a hard statistical read, a research-grade engagement runs a higher N.
- **Methodology version:** the documented procedure the runs follow.
- **Mission shape:** the declared question, the denominators, and the claim classes (see [Pre-registered protocol](#pre-registered-protocol)).

A study always declares which panel version it ran on. **Same panel version \+ same mission = comparable runs.** That is the contract.

<Note>
  The panel is composed against the mix of assistants real shoppers use, not a single model. When that mix shifts, the panel moves with it. The next two sections cover how.
</Note>

## Version pinning

Every run is stamped at dispatch with:

- the **methodology version and a hash of the exact methodology document** the run followed,
- the **harness version** that produced it, and
- a **hash of the exact brief** the agents received.

These stamps travel with the run through to the report. Any number in any report can be traced back to which instrument produced it, and a re-run on the same stamps is a true re-measurement, not an impression.

The methodology stamp is checked against the procedure actually loaded on the machine at run time; a mismatch is caught rather than absorbed.

## Drift, deprecation, and what counts as a change

Two different things can change, and we treat them differently.

<CardGroup cols={2}>
  <Card title="We change the panel" icon="git-branch">
    Composition, lanes, weights, or methodology. That is a **new panel version**: versions are numbered, the prior version's definition is frozen, and a study declares which one it ran on. A version change is surfaced before the next run executes on it, and where providers still serve the older models, a **bridge run** (the same mission on both panel versions) keeps a trendline across the transition.
  </Card>

  <Card title="The world changes under a pinned panel" icon="radar">
    A provider updates a model behind its public name. The panel measures the **live consumer mix**: when the models people actually use shift, that movement is signal, not noise. The served model identity is recorded per trip, so a material provider-side shift is surfaced in the run's findings rather than absorbed.
  </Card>
</CardGroup>

The served model identity is recorded per trip rather than hard-pinned to a frozen snapshot, because a hard-pinned panel would diverge from what real shoppers experience. The discipline is **record-and-disclose**: the panel tracks the world, and the seam is recorded in the run.

**Deprecation.** When a provider retires a model, that deprecation is treated as a panel change: a new numbered version, with the bridge-run protocol offered where it is feasible. Because every run is stamped with its panel version (see [Version pinning](#version-pinning)), panel history stays interpretable: any past report can still be read against the exact version it ran on.

## Pre-registered protocol

Missions, denominators, and claim classes are declared **before** the run. [Counts come before rates](/insights/reading-a-report/count-ledger); every claim carries its [evidence reference](/insights/reading-a-report/receipts-and-proof); the claim boundaries print on the report. The report generator cannot change them after seeing results.

## Stability protocol

This is the protocol the instrument follows, not a machine-guaranteed property of every run. The discipline is stated here; the specifics travel with each study.

By design, a panel version remains the panel-of-record for a stated window unless a provider deprecation forces a change. Baseline re-runs are meant to execute on the original panel version where providers still serve it, with the bridge protocol applying otherwise. And if a [receipt](/insights/reading-a-report/receipts-and-proof) doesn't hold up on re-inspection, the study re-runs rather than standing on a number it can no longer defend.

## Short answers

<AccordionGroup>
  <Accordion title="Can't you just tweak the panel until it says what someone wants?">
    No. Composition is locked per version, the stamp records the instrument's identity, and the protocol is pre-registered. A tweaked panel is a _different, numbered instrument_, not a quiet adjustment.
  </Accordion>

  <Accordion title="What happens when the next major model ships?">
    If it changes what consumers' agents actually are, the panel version increments to match the world, with notice given and a bridge run offered, so a trendline shows the seam rather than hiding it.
  </Accordion>

  <Accordion title="How do I audit this?">
    Every report's receipts carry the stamps. Ask for any run's evidence; every number traces back to its stamps.
  </Accordion>
</AccordionGroup>
